# AGENTS.md

## 项目定位
- 本目录是“基于 Vue3 + SpringBoot 的博物馆文物综合管理平台”后端子项目。
- 这是毕业设计项目，优先保证功能可运行、演示稳定、逻辑清晰、易于答辩。
- 不要过度设计，优先选择简单、直接、容易维护的实现方案。

## 后端技术栈
- Java 17
- Spring Boot 3
- Maven
- MyBatis-Plus
- Spring Security
- JWT
- MySQL 8
- Redis
- Lombok

## 作用范围
- 只允许读取、创建、修改、删除 `mhmp_back` 目录内的文件。
- 不要修改前端目录 `mhmp` 中的任何文件。
- 除非用户明确说明，否则不要跨目录修改文件。
- 如果需求看起来需要前端配合，也只能处理后端接口、返回结构、业务逻辑，不要直接改前端文件。

## 后端规则
- 以现有 `mhmp_db` 数据库为准，不重新设计数据库。
- 不要随意修改表结构、索引、约束、初始化数据。
- 如果确实必须改数据库，请先说明原因并等待确认。
- 统一使用 `Result` 作为返回结构。
- 统一使用全局异常处理；除非有明确业务需要，不要到处写零散 `try/catch`。
- `deleted` 为逻辑删除字段，默认优先逻辑删除，不直接物理删除。
- 按现有分层结构开发：`controller / service / serviceImpl / mapper / entity / dto / vo / config / common`。
- `controller` 只处理请求接收、参数校验、结果返回。
- 业务逻辑放在 `service` 和 `serviceImpl`。
- 数据访问逻辑放在 `mapper`。
- 请求参数优先使用 `dto`，响应数据优先使用 `vo`，不要随意和 `entity` 混用。

## 权限与认证规则
- 权限体系基于 `sys_user`、`sys_role`、`sys_menu`、`sys_user_role`、`sys_role_menu`。
- 权限标识优先使用 `permission_code`。
- 认证与鉴权遵循现有 Spring Security + JWT + Redis 方案，不要更换整体认证设计。
- 新增受保护接口时，权限命名、鉴权流程、返回风格要与现有项目保持一致。

## 实现优先级
- 优先复用现有代码风格、目录结构和已有能力。
- 修改范围尽量小，不要顺手重构无关模块。
- 不要引入微服务、分布式架构或其他过重方案。
- 除非用户明确接受，否则不要破坏已有接口兼容性。
- 代码注释适量，重点说明业务规则和不易看懂的逻辑。
- 优先实现“容易演示、容易讲清楚、稳定可运行”的方案，而不是理论上更复杂的最优解。

## 工作方式
- 修改前先分析相关后端代码、调用链和依赖关系。
- 需求不明确时，优先选择与当前项目风格一致、最简单可运行的后端实现。
- 如果发现已有改动与当前任务冲突，不要擅自回退，先判断影响范围。
- 命名风格、返回格式、异常处理风格、权限风格要与现有后端保持一致。
